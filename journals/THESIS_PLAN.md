## 1. План работы

### Этап 1. Подготовка теоретической базы (Глава 1)

1. Сформулировать проблематику автоматизации производства караоке-контента в условиях клуба: объём ручного труда, длительность подготовки, стоимость, отсутствие быстрой реакции на пользовательский трек.
2. Описать типовой профиль нагрузки караоке-сервиса: всплески запросов в вечерние часы, длительная обработка одного запроса (минуты), требования к фоновой обработке без блокировки пользовательского интерфейса.
3. Описать жизненный цикл караоке-трека и обосновать декомпозицию обработки на изолированные этапы.
4. Провести обзор современных методов и моделей по каждому этапу:
   - источниковое разделение (MDX-Net, Demucs, BS-Roformer, Mel-Band RoFormer, Open-Unmix);
   - детектирование голосовой активности (Silero VAD, WebRTC VAD, RMS-методы);
   - автоматическое распознавание речи (Whisper, WhisperX, faster-whisper, MMS, wav2vec 2.0);
   - поиск текстов песен (Genius, LRCLIB, MusicBrainz, агентные подходы с LLM, метапоисковые системы);
   - принудительное выравнивание (Montreal Forced Aligner, CTC-методы, MMS-300m, WhisperX align);
   - сегментация на строки и эвристики паузной разметки.
5. Провести обзор архитектурных подходов к построению высоконагруженных систем обработки медиаконтента:
   - паттерны асинхронных конвейеров (pipeline pattern, stage decomposition);
   - очереди сообщений (RabbitMQ, Kafka, Redis Streams) и сравнительный анализ под задачу;
   - объектные хранилища (S3-совместимые) для тяжёлых артефактов;
   - стратегии масштабирования воркеров (horizontal scaling, work stealing, sharding по GPU);
   - паттерны устойчивости (retry, circuit breaker, bulkhead, dead-letter queue, idempotency keys);
   - наблюдаемость (структурированное логирование, метрики, трассировки).
6. Проанализировать коммерческие решения (Karafun, Sonoix, MVSEP, LyricsGenerator) и open-source проекты (UltraStarManager, Karaoke Mugen) — преимущества, ограничения, применимость.
7. Сформулировать функциональные и нефункциональные требования к подсистеме (включая SLA по времени обработки, целевой throughput, требования к доступности и устойчивости).
8. Сформулировать цели, задачи и научную новизну работы.

### Этап 2. Проектирование архитектуры подсистемы (Глава 2)

1. Спроектировать место worker-подсистемы в общей архитектуре караоке-приложения (взаимодействие с backend, message broker, объектным хранилищем, реляционной БД).
2. Применить C4-модель для документирования архитектуры (Context, Container, Component diagrams).
3. Декомпозировать пайплайн обработки на изолированные шаги (разделение → VAD → ASR → поиск текста → выравнивание → разбиение на строки → финализация).
4. Обосновать выбор моделей для каждого этапа на основании сравнительных характеристик (SDR, WER, лицензирование, требования к ресурсам).
5. Спроектировать схему асинхронного межсервисного взаимодействия через RabbitMQ:
   - три обмена (jobs, job.progress, rec) с разной семантикой (direct/fanout);
   - prefetch_count=1 для равномерного распределения по воркерам;
   - manual ack как механизм гарантии доставки;
   - DLQ как защита от отравленных сообщений.
6. Спроектировать модель данных задач (`job_queue` с JSONB-полем для промежуточных результатов) — поддержка возобновления и инспекции состояния.
7. Спроектировать управление вычислительными ресурсами GPU:
   - резидентное размещение моделей в VRAM между задачами для устранения cold start;
   - параллелизация ввода-вывода (загрузка/выгрузка S3, конвертация ffmpeg) с вычислениями (asyncio.gather, фоновые задачи);
   - изоляция CTC-инференса в подпроцессе для защиты основного процесса от утечек VRAM и heap corruption;
   - переключение между FP16/FP32, autocast, чанковая обработка с overlap-add.
8. Спроектировать стратегию устойчивости и обработки отказов:
   - повторные попытки с экспоненциальной задержкой;
   - dead-letter queue для треков, требующих ручного разбора;
   - идемпотентность на уровне ключей S3 и записей `job_queue`;
   - graceful shutdown с сохранением текущего шага в БД.
9. Спроектировать систему наблюдаемости:
   - структурированное логирование через structlog (JSON-выход);
   - публикация прогресса через fanout-обмен в SSE-канал;
   - метрики времени по шагам пайплайна.
10. Описать ожидаемый профиль нагрузки и стратегию горизонтального масштабирования (несколько воркеров на одну GPU, multi-GPU развёртывание).

### Этап 3. Реализация компонентов подсистемы (Глава 3)

1. Реализовать модуль источникового разделения (BS-Roformer): batched chunk processing, overlap-add реконструкция, FP16 inference, autocast.
2. Реализовать дополнительный шаг разделения вокала на ведущий и фоновый (Mel-Band RoFormer aufr33), обосновать необходимость для повышения качества последующих этапов.
3. Реализовать модуль детектирования голосовой активности на чистом PyTorch (без librosa) — `torch.unfold` + RMS + threshold.
4. Интегрировать модуль распознавания речи на основе HuggingFace Transformers Whisper с резидентным размещением модели в GPU-памяти между задачами.
5. Реализовать многопровайдерный поиск текстов песен (provider chain): Genius, LRCLIB, lyricsovh, chartlyrics, simpmusic.
6. Реализовать резервный агентный контур поиска: LLM (DeepSeek) с инструментами `web_search` (Yandex Search API) и `fetch_webpage`, локальный SearXNG как метапоисковый прокси.
7. Реализовать многоступенчатый алгоритм скоринга и сопоставления текстов (`normalizer`, `expander`, `linguistics`, `matcher`, `scorer`) — устойчивое сопоставление кандидатов с ASR-выводом.
8. Реализовать модуль принудительного послогового выравнивания: MMS-300m CTC через `torchaudio.functional.forced_align` + `merge_tokens`, изоляция в подпроцессе.
9. Реализовать постобработку таймингов: предобрезка интро через Silero VAD, три RMS-пасса (коррекция начала строк, окончания слов, удержания согласных).
10. Реализовать адаптивный алгоритм разбиения текста на строки с автоматическим выбором между паузной и битовой эвристикой (librosa.beat.beat_track).
11. Реализовать управление состоянием задач, конвейер прогресса, финализацию треков и публикацию событий для смежных подсистем.
12. Реализовать инструментирование наблюдаемости: structlog в JSON-формате, отметки времени по шагам, агрегированные метрики.

### Этап 4. Экспериментальное исследование и нагрузочные испытания (Глава 4)

1. Сформулировать методологию оценки: тестовая выборка треков (русский/английский, разные жанры и сложности), эталонные метрики, инструменты воспроизведения.
2. Провести сравнительные эксперименты по моделям разделения источников (MDX-NET vs BS-Roformer vs Mel-Band RoFormer): SDR, время обработки, качество отделения вокала на сложных кейсах.
3. Провести сравнительные эксперименты по методам послогового выравнивания (proportional split, WhisperX force_align, CTC word-level, CTC char-level, CTC Hybrid и др.): MAE, hit rate (<0.1с), время обработки, требования к ресурсам.
4. Оценить качество поиска текстов и метаданных (точность определения исполнителя/названия, доля треков с найденным текстом, влияние ASR-подсказок).
5. **Провести нагрузочные испытания подсистемы**:
   - end-to-end время обработки трека (p50, p95, p99);
   - throughput воркера (треки/минута) при разной интенсивности входного потока;
   - устойчивость при превышении пропускной способности (рост глубины очереди, поведение backpressure);
   - поведение при сбоях — отказ воркера в середине задачи, недоступность S3, недоступность LLM-провайдера;
   - время восстановления после рестарта воркера.
6. Оценить эффективность горизонтального масштабирования:
   - утилизация GPU при 1, 2, 3 воркерах на одну карту;
   - масштабирование на multi-GPU (опыт бутстрапа на 8×RTX 4090, 24 воркера, 17 315 треков за ~19 часов);
   - закон Амдала и точка насыщения.
7. Оценить эксплуатационные характеристики: потребление VRAM/RAM, нагрузка на CPU и сеть, влияние холодного старта моделей.
8. Провести качественную оценку результатов на тестовой выборке: визуальный анализ синхронизации подсветки, выявление типичных ошибок, ограничения системы.
9. Сопоставить полученные показатели с заявленными требованиями SLA и сформулировать выводы о применимости решения и направления дальнейшего развития.

### Этап 5. Оформление и подготовка к защите

1. Написать введение и заключение в соответствии с требованиями ВШЭ и методичкой программы.
2. Подготовить иллюстративный материал: C4-диаграммы, диаграммы последовательности, графики нагрузочных испытаний, сравнительные таблицы метрик, скриншоты пользовательского интерфейса с послоговой подсветкой.
3. Сформировать список источников (ориентировочно 70–90 наименований: научные статьи по MIR, ASR, source separation, forced alignment; работы по распределённым системам, очередям сообщений, паттернам отказоустойчивости; документация инструментов и моделей).
4. Подготовить приложения: листинги ключевых алгоритмов, расширенные таблицы экспериментов, протоколы нагрузочных испытаний, акт о внедрении.
5. Подготовить доклад (до 20 минут) и презентацию для защиты.

## 3. Краткое обоснование выбора плана и оглавления

### Соответствие профилю программы

Программа «Проектирование и разработка высоконагруженных информационных систем» — практико-ориентированная, с партнёрами 1С, VK, Газпром Нефть. Тематика программы включает горизонтальное масштабирование, асинхронность, кэширование, мониторинг, отладку, профилирование, отказоустойчивость и оптимизацию. Поэтому в плане сделан явный акцент именно на high-load-составляющую: подсистема воркера представлена как асинхронный конвейер инференса ML-моделей, в котором сами ML-модели — лишь полезная нагрузка, а основной инженерный вклад приходится на архитектуру очередей, изоляцию ресурсов, управление состоянием, устойчивость и масштабирование. Этот сдвиг отражён в новых разделах 1.4, 2.5–2.8, 3.7.4, 4.5–4.7.

Если бы план был ориентирован только на «качество ML-компонентов» (как часто делают на ПМИ), у комиссии возник бы вопрос «почему эта работа защищается на программе high-load систем». Текущая структура отвечает на этот вопрос напрямую.

### Тип ВКР: проектно-аналитический с экспериментальной компонентой

Согласно методическим рекомендациям ВШЭ, магистерская диссертация может относиться к научно-исследовательскому или проектно-аналитическому типу. Для практико-ориентированной программы high-load систем с её партнёрством с индустрией более уместен **проектно-аналитический тип**: вы спроектировали и реализовали реальную подсистему, а не просто провели сравнительное исследование. Однако без экспериментальной главы 4 проектная ВКР воспринимается как «отчёт о работе» — поэтому обязательно сохранена экспериментально-исследовательская компонента (сравнительные эксперименты + нагрузочные испытания), что переводит работу в строгий формат «спроектировал → реализовал → измерил → доказал применимость».

### Почему четыре главы, а не три

Стандарт ВШЭ допускает 3–5 глав основной части. Для вашего материала четыре главы оптимальны:

- В тему «автоматическая обработка контента» входит шесть содержательно различных этапов (разделение, VAD, ASR, поиск текста, выравнивание, разбиение на строки) плюс высоконагруженная инфраструктура. Сводить проектирование и реализацию в одну главу — значит делать главу непропорционально большой (25–30 страниц) и терять структурную чёткость.
- Разделение «Глава 2 — проектирование» и «Глава 3 — реализация» удобно для защиты: рецензент сначала видит обоснованные архитектурные решения, потом проверяет их воплощение. Это снижает риск претензий типа «реализация не следует из проектных решений».
- Отдельная экспериментальная Глава 4 необходима программе high-load: без нагрузочных испытаний работа не может считаться полноценной для специальности. Это критическое требование, которое часто проваливают студенты, сосредоточившись только на функциональности.

При 4 главах объём 60–80 страниц распределяется естественно: введение 6–8 страниц, главы по 12–16 страниц каждая, заключение 3–4 страницы.

### Почему рекомендательная подсистема исключена даже из обзора

Вы предупредили, что её делал другой автор. По требованиям ВШЭ ВКР должна быть личным вкладом автора, иначе на защите неизбежен вопрос «что именно сделали вы». Поэтому в плане нет ни главы по рекомендациям, ни их компонентов. В Главе 2 (архитектурный контекст) рекомендательная подсистема упоминается только как смежный потребитель сообщений из RabbitMQ — это необходимо для целостности картины, но не выносится в самостоятельный раздел.

### Что нового в Главе 1 по сравнению с «общей» структурой

Раздел 1.4 (архитектурные подходы к построению высоконагруженных систем) — это специальный обзор для программы high-load. Без него теоретическая база работы окажется однобокой (только ML), и комиссия не увидит, что вы знаете предметную область программы. В разделе обязательно сравнить RabbitMQ с альтернативами (Kafka, Redis Streams, NATS) и обосновать выбор — это типичный вопрос на защите.

### Что нового в Главе 2 по сравнению с «обычной» проектной работой

- Раздел 2.4.2 (семантика доставки) и раздел 2.6 (устойчивость) — это специальные темы для high-load: prefetch_count, manual ack, durable queues, DLQ, идемпотентность. Без них работа не воспринимается как серьёзная high-load инженерия.
- Раздел 2.5 (управление вычислительными ресурсами) — главная сильная сторона вашей реализации: резидентное размещение моделей в VRAM, asyncio.gather для параллелизации, изоляция CTC в подпроцессе, чанковая обработка с overlap-add. Если их не выделить, в защите они растворятся внутри описания шагов.
- Раздел 2.7 (наблюдаемость) — без него любая high-load работа выглядит незавершённой. У вас есть structlog, прогресс через fanout, метрики времени — это нужно явно представить.
- Раздел 2.8 (горизонтальное масштабирование) — обязательный раздел для программы high-load. У вас есть реальный опыт multi-GPU bootstrap (8×RTX 4090, 24 воркера), это сильный материал.

### Почему Глава 4 расширена нагрузочными испытаниями

Это критически важное изменение по сравнению с первой версией плана. На программе high-load систем работа без нагрузочных испытаний не считается полноценной. Конкретно:

- **4.5.2 (p50/p95/p99 latency)** — стандарт de facto в индустрии для оценки производительности.
- **4.5.3–4.5.4 (throughput, backpressure)** — показывает, что вы понимаете, как ведёт себя система при превышении мощности.
- **4.5.5 (закон Амдала, точка насыщения)** — формальная оценка эффективности масштабирования.
- **4.5.6 (опыт массового бутстрапа)** — у вас реально обработано 17 315 треков на 8×RTX 4090 за ~19 часов. Этот опыт — готовый кейс для нагрузочной оценки, его нельзя терять.
- **4.6 (устойчивость к отказам)** — доказательство, что система действительно отказоустойчива, а не только заявляет это.

### Почему такой состав введения

Все десять элементов введения (актуальность, объект, предмет, цель, задачи, новизна, положения на защиту, методология, теоретическая и практическая значимость, апробация) — это формальное требование ВШЭ к магистерским работам. Любой пропуск приводит к замечанию на нормоконтроле. Объём введения — около 6–8 страниц (≈10% работы), что соответствует методическим рекомендациям.

### Почему такой набор приложений

- Приложение А (API подсистемы) — выносит формальные спецификации из основного текста, экономит объём.
- Приложение Б (расширенные таблицы) — основная часть содержит сводные таблицы, полные данные размещаются в приложениях.
- Приложение В (протоколы нагрузочных испытаний) — обязательно для работы по high-load: исходные данные нагрузочных тестов, конфигурации, графики.
- Приложение Г (листинги) — обязательное требование для проектных ВКР по техническим направлениям.
- Приложение Д (скриншоты) — визуальное доказательство того, что система работает в продуктовом интерфейсе.
- Приложение Е (акт о внедрении) — даёт дополнительные баллы при наличии и снимает вопросы о практической значимости. Особенно ценно для программы с партнёрами из индустрии.

**Оглавление** — стандартное: перечень всех разделов с номерами страниц, оформляется по общему образцу ВШЭ. Конкретного «программного шаблона» в открытом доступе нет — используйте общий образец из методических указаний (см. ниже).

### 5.3. Конкретные требования к объёму, шрифтам, полям, межстрочным интервалам

**Объём.** Магистерская ВКР — **не более 75 страниц** основного текста без приложений (методичка ВШЭ, февраль 2024). Ориентир для технической ВКР: 60–75 страниц основной части. Введение и заключение — по 1–3 страницы каждый (в магистратуре допустимо до 6–8 страниц, особенно введение). Источник: [методичка ВКР НИУ ВШЭ февраль 2024](https://www.hse.ru/data/2024/02/16/2096018881/Метод.рекомендации%20ВКР_%20У_февраль_24.pdf), методичка ВКР направления 09.03.01 «Информатика и вычислительная техника» НИУ ВШЭ — СПб (бакалавриат, но даёт ориентир на технические требования СПб).

**Шрифт и интервалы:**
- Шрифт: **Times New Roman**.
- Размер: **14 кегль** (методичка магистерских ВКР ВШЭ 2024). Замечание: в более старом общеуниверситетском документе «Требования к оформлению» 2020 года указан 12 кегль, но для магистратуры в актуальной методичке принят 14.
- Межстрочный интервал: **1,5**.
- Примерное число знаков на странице: **~2000**.
- Каждый абзац — с красной строки.
- Каждая глава, введение, заключение, библиография, приложения — **с новой страницы**.

**Поля страницы:**
- Левое поле: **35 мм**.
- Правое поле: **не менее 10 мм** (методичка магистерских ВКР 2024) или **не менее 15 мм** (общие требования к оформлению — старее).
- Верхнее и нижнее поля: **не менее 20 мм**.

**Нумерация страниц:** сквозная, начиная с титульного листа, но номер на титульном листе не печатается. Формат — A4.

Источник: [методичка ВКР НИУ ВШЭ февраль 2024](https://www.hse.ru/data/2024/02/16/2096018881/Метод.рекомендации%20ВКР_%20У_февраль_24.pdf), стр. 6–7; [Требования к оформлению курсовых и ВКР НИУ ВШЭ](https://www.hse.ru/data/2020/02/18/1575426747/Требования%20к%20оформлению%20курсовых%20работ%20и%20ВКР.pdf).

### 5.4. Требования к списку источников (ГОСТ vs APA vs IEEE)

**Ответ:** для **технических ВКР НИУ ВШЭ — СПб** список источников оформляется по **ГОСТ Р 7.0.100‑2018** «Библиографическая запись. Библиографическое описание. Общие требования и правила составления». Сама работа оформляется по **ГОСТ 7.32‑2017** «СИБИД. Отчёт о научно-исследовательской работе. Структура и правила оформления». Сокращения — по **ГОСТ Р 7.0.12‑2011** (русский) и **ГОСТ 7.11‑2004** (иностранный).

**Цитирование в тексте** — по ГОСТу, в квадратных скобках с номером источника и (при необходимости) страницы: `[12, с. 25]`. Это техника, принятая для технических работ в ВШЭ — СПб. У гуманитарных программ ВШЭ встречается APA-стиль `(Witztum, 1998, p. 25)`, но для технической программы это нехарактерно.

**Структура списка:**
1. Литература на русском языке — в алфавитном порядке.
2. Литература на иностранных языках — в алфавитном порядке (после русской).
3. Источники данных, нормативные документы, интернет-источники — в хронологическом порядке (или по типу).

В список включаются только источники, на которые есть ссылки в тексте.

Источник: [методические рекомендации по подготовке и защите ВКР НИУ ВШЭ — СПб (направление 09.03.01)](https://spb.hse.ru/mirror/pubs/share/1080744939); [страница «Bibliographic Standards» НИУ ВШЭ](https://academics.hse.ru/bibliography/standards).

### 5.5. Обязательные разделы введения

**Ответ:** для технических ВКР НИУ ВШЭ — СПб введение должно содержать:

1. **Актуальность темы** (включая степень научной разработанности).
2. **Объект и предмет** работы.
3. **Цель и задачи** работы (задачи определяются исходя из темы).
4. **Теоретические основы**, методология/методика, технические средства.
5. **Теоретическая и/или практическая значимость** работы.
6. **Научная новизна и апробация** работы.
7. **Характеристика структуры** работы.

Для **магистерской** ВКР дополнительно желательны:
- **Положения, выносимые на защиту** (3–5 пунктов).
- **Гипотеза исследования** (по согласованию с научным руководителем).

Объём введения — рекомендуется до 3 страниц для бакалавриата, для магистратуры допустимо 5–8 страниц (≈10% объёма работы). У моего текущего оглавления введение содержит все эти элементы.

Источник: [методичка ВКР НИУ ВШЭ — СПб направления 09.03.01](https://spb.hse.ru/mirror/pubs/share/1080744939), раздел 3 «Содержание и объём ВКР».

### 5.6. Доля иностранных источников и доля источников за последние 5 лет

**Доля иностранных источников.** Для бакалавриата по направлению 09.03.01 ВШЭ — СПб требует **не менее 10 источников на иностранных языках** при общем объёме списка ≥45. Это даёт ориентир ≈22% иностранных. Для магистерской работы технического профиля разумно держать долю **30–40% иностранных источников** — это типичная норма для научных ВКР в области CS, и для high-load темы она достижима естественно (большая часть качественных источников по очередям сообщений, source separation, ASR — на английском).

**Доля свежих источников.** Прямой нормы «N% за последние 5 лет» в открытых документах ВШЭ — СПб для технических ВКР я не нашёл. Однако:
- В методичке ВКР ВШЭ 2024 (для социальных программ) указано: в обзоре литературы рассматриваются «только те исследования, результаты которых были опубликованы в признанных научных журналах **не ранее, чем за 6 лет до момента защиты диссертации**» — т.е. последние 6 лет. Это можно использовать как ориентир.
- Общая практика российских вузов: **не менее 50–60% источников за последние 5 лет**.

**Общий объём списка для магистратуры:** ориентир 60–90 источников. Для бакалавриата технических ВКР НИУ ВШЭ — СПб минимум 45.

Источник: [методичка ВКР НИУ ВШЭ — СПб направления 09.03.01](https://spb.hse.ru/mirror/pubs/share/1080744939); [методичка ВКР НИУ ВШЭ февраль 2024](https://www.hse.ru/data/2024/02/16/2096018881/Метод.рекомендации%20ВКР_%20У_февраль_24.pdf).

### 5.7. Требования к проверке на антиплагиат

**Ответ:** проверка через систему «Антиплагиат.ВУЗ» **обязательна** для всех ВКР НИУ ВШЭ. Финальная версия загружается студентом в LMS, оттуда автоматически уходит на проверку. Менеджер программы получает отчёт.

**Порог:** **доля оригинальности ≥80%**, то есть **доля заимствований <20%**. Если оригинальность падает ниже 80%, работа в обязательном порядке направляется руководителю и декану факультета — это сигнал «повышенного внимания», но не автоматическое отклонение. Любая доля **плагиата** (неоформленных заимствований) недопустима — это нарушение академических норм, которое может привести к оценке «0/10» и дисциплинарному взысканию вплоть до отчисления.

**Что считается заимствованием:** совпадение с любым текстом из базы «Антиплагиата» (интернет, корпоративная база ВШЭ из работ прошлых лет, научные публикации). 20% — это норма для **правильно оформленных цитат** с указанием авторов и источников.

**Что нельзя:** использование генеративных ИИ для написания текста без декларирования (с мая 2024 действует отдельный регламент о декларации использования ИИ при подготовке учебных работ). Вместе с ВКР загружается декларация об использовании / неиспользовании ИИ.

**Лайфхак для студентов:** через ЕЛК доступна бесплатная предварительная проверка в системе «Антиплагиат» — до 15 загрузок в год. Прогоните черновик заранее, чтобы не получить сюрприз перед сроком.

Источник: [Памятка для студентов «Проверка работ на плагиат» НИУ ВШЭ — СПб](https://spb.hse.ru/mirror/pubs/share/463734408.pdf); [страница «Антиплагиат» НИУ ВШЭ](https://www.hse.ru/org/hse/antiplagiat_info/); [Регламент проверки на плагиат и использование ИИ от 8 мая 2024](https://www.hse.ru/docs/922831988.html).

## 5а. Как эти ответы повлияли на план

Большинство параметров уже учтены в моей текущей структуре. Что стоит подправить при написании:

1. **Шапка работы** — на титульном листе использовать формулировку из п. 5.1.
2. **Введение** — теперь явно перечислены семь обязательных пунктов СПб + два магистерских (положения на защиту, гипотеза). В оглавлении они есть.
3. **Список литературы** — целиться в 60–90 источников, ≥30% иностранных, оформление по ГОСТ Р 7.0.100‑2018, цитирование в квадратных скобках `[N, с. M]`.
4. **Антиплагиат** — держать оригинальность ≥80%. Длинные технические описания моделей (BS-Roformer, MMS-300m) и API стоит давать своими словами с явными ссылками на оригинальные публикации, а не копировать формулировки из карточек моделей.
5. **Шрифт/поля** — TNR 14, 1.5 интервал, левое поле 35 мм, правое ≥10 мм, верх/низ ≥20 мм. Каждый структурный элемент — с новой страницы.
6. **Объём** — целиться в 60–75 страниц основного текста (без приложений). Приложения — по необходимости.
7. **Аннотация** — обязательно русская и английская версии, до 1500 знаков с пробелами, с ключевыми словами.

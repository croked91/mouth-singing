## Фаза 15: Интеграционное тестирование и hardening

### Входные артефакты
- Результат Фазы 14 (полностью развёрнутое приложение)
- `journals/ARCHITECTURE.md` — все диаграммы взаимодействия (раздел 6) как сценарии для E2E тестов
- Все `design/prompts/` для UI-верификации

### Задачи фазы

#### Оркестратор (ты)
Финальная фаза. Запускаешь `polyglot-test-engineer` для полного E2E прогона. По результатам — `python-developer` и `frontend-web-client` для фиксов. В конце — `software-architect` для финального ревью кодовой базы.

#### Подагент `polyglot-test-engineer`
Проводит полное тестирование:

1. **E2E сценарий** (воспроизводит все 5 диаграмм взаимодействия из ARCHITECTURE.md):
   - Создание сессии (room_id=101) → добавление 3 участников (Вася, авто-ник, Маша)
   - Поиск трека по тексту → выбор → добавление в очередь
   - Получение рекомендаций (popular для нового участника)
   - Загрузка MP3 → SSE прогресс → клип готов → добавление в очередь
   - Start → воспроизведение (плеер загружается) → Finish → рекомендации обновились (strategy=last)
   - Skip → участник перемещается в конец
   - Admin PIN → завершение сессии → возврат на WelcomePage

2. **Стресс-тест**:
   - 5 участников в одной сессии
   - 20 треков в очереди
   - 3 одновременные загрузки MP3
   - SSE для всех 3 загрузок параллельно

3. **Edge cases**:
   - Невалидный MP3 (txt файл с расширением .mp3)
   - Пустой каталог (нет треков) → рекомендации "popular" пустые
   - Два одновременных finish (race condition)
   - Сессия с 0 участниками → попытка добавить трек в очередь
   - Потеря SSE-соединения и переподключение

4. **Проверка логирования**: все сервисы пишут структурированные JSON-логи с контекстом (session_id, track_id, job_id).

#### Подагент `python-developer`
Фиксит баги, найденные при тестировании (бэкенд).

#### Подагент `frontend-web-client`
Фиксит баги, найденные при тестировании (фронтенд).

#### Подагент `software-architect`
Финальное ревью:
- Нет дублирования кода между backend/worker/bootstrap (всё через shared/)
- Нет утечек ресурсов (незакрытые соединения, файловые дескрипторы)
- Корректная обработка ошибок во всех сервисах
- Соответствие ADR

#### Пользователь
Проводит финальное ручное тестирование полного сценария. Подтверждает готовность приложения.

### Выходные артефакты
- E2E тесты проходят без ошибок
- Стресс-тест: стабильно при 5 участниках, 20 треках, 3 одновременных загрузках
- Все edge cases обработаны
- Структурированные логи во всех сервисах
- Журналы проекта актуальны (PROJECT_LOG, PHASES)
- Финальный коммит

# Экспертный обзор системы рекомендаций

**Дата:** 2026-03-19
**Статус:** На обсуждении
**Участники:** 7 экспертных подагентов (пользователь-одиночка, малая группа, большая группа, эксперт по музыке, эксперт по рекомендательным системам, архитектор, UX-дизайнер)

---

## Резюме

Система рекомендаций **технически грамотна и хорошо спроектирована** для базового сценария. 4-уровневый каскад стратегий (POPULAR -> LAST -> LAST_TWO_AVG -> SESSION_AVG), fusion KNN по двум пространствам (аудио 70% + лирика 30%), EMA-портрет и граф переходов -- это сильная основа, которую не имеют многие коммерческие караоке-системы.

Однако все эксперты сходятся в том, что **система умнее, чем выглядит**: мощный алгоритм скрыт за плоским UI без обратной связи с пользователем. Основные зоны роста: diversity, UX рекомендаций, групповые сценарии и несколько алгоритмических багов.

## 1. АЛГОРИТМИЧЕСКИЕ БАГИ И ФИКСЫ (делать в первую очередь)

### 1.1 Bias при fusion scoring для треков без лирики
**Источник:** Эксперт по рекомендательным системам
**Суть:** Кандидат без lyrics_vector получает `lyrics_score = 0.0`, что систематически занижает его fused score. Инструментальный трек с `audio_score = 0.95` получит `fused = 0.665`, а трек с лирикой и `audio_score = 0.9, lyrics_score = 0.8` -- `fused = 0.87`.
**Фикс:** При отсутствии lyrics у кандидата: `fused = audio_score` (без penalty). Формально: нормализовать по доступным модальностям.
**Сложность:** Низкая (~5 строк)

### 1.2 Race condition в record_transition
**Источник:** Архитектор, Эксперт по рекомендательным системам
**Суть:** Read-modify-write weight в QDrant без блокировки. Два concurrent finish_entry для одной пары треков могут потерять инкремент.
**Фикс:** Перенести transitions в SQLite с `UPDATE weight = weight + 1` (атомарно), или использовать QDrant set_payload с conditional update.
**Связано с:** п. 1.3

### 1.3 Transitions хранятся в QDrant, хотя используются как key-value
**Источник:** Архитектор, Эксперт по рекомендательным системам
**Суть:** Коллекция `transitions` хранит 45-D вектор, который **никогда не используется** для KNN. `scroll_filtered` делает payload-фильтрацию, а не similarity search. Вектор существует только потому, что QDrant требует его при upsert.
**Рекомендация:** Перенести transitions в SQLite-таблицу `(from_track_id, to_track_id, weight)` с составным индексом. Устранит race condition, упростит бэкапы, уберёт лишний asyncio.to_thread.
**Сложность:** Средняя

### 1.4 N+1 запрос в semantic search
**Источник:** Архитектор
**Суть:** В `_semantic_search` каждый hit делает отдельный `get_track(point_id)`. Метод `get_tracks_by_ids()` уже существует.
**Фикс:** Заменить цикл на batch-запрос.
**Сложность:** Низкая

### 1.5 Sequential QDrant-вызовы вместо parallel
**Источник:** Архитектор, Эксперт по рекомендательным системам
**Суть:** `_last_two_avg_strategy` делает 4 последовательных retrieve. `_last_strategy` -- 2 последовательных. Все независимы и могут идти через `asyncio.gather()`.
**Фикс:** Обернуть в gather.
**Сложность:** Низкая

### 1.6 Отсутствие timeout у QdrantClient
**Источник:** Архитектор
**Суть:** Нет timeout при создании QdrantClient. Если QDrant зависнет -- request заблокирует поток навсегда.
**Фикс:** Добавить `timeout=10` в конструктор.
**Сложность:** Минимальная

## 2. DIVERSITY И FILTER BUBBLE (высокий приоритет)

### 2.1 Отсутствие MMR (Maximal Marginal Relevance)
**Источник:** Эксперт по рекомендательным системам
**Суть:** KNN возвращает top-N по fused score без diversity re-ranking. 10 рекомендаций могут быть 10 каверов одной песни.
**Рекомендация:** Greedy MMR re-ranking с `lambda = 0.7` (~10 строк кода):
```python
selected = []
for _ in range(limit):
    best = max(candidates, key=lambda c:
        0.7 * c.score - 0.3 * max_sim(c, selected))
    selected.append(best)
    candidates.remove(best)
```
**Влияние:** Существенно повышает разнообразие рекомендаций
**Сложность:** Низкая

### 2.2 Нет diversity injection в SESSION_AVG
**Источник:** Пользователь-одиночка, Малая группа
**Суть:** В стратегии POPULAR есть 30% random (хорошо!), но в SESSION_AVG -- 0%. После 5+ треков система замыкается в "пузыре". Одиночка особенно уязвим.
**Рекомендация:** Подмешивать 10-15% exploration-треков из далёкого кластера в стратегиях LAST/LAST_TWO_AVG/SESSION_AVG.
**Сложность:** Низкая-Средняя

### 2.3 Popularity bias в графе переходов
**Источник:** Эксперт по рекомендательным системам, Эксперт по музыке
**Суть:** Weight как raw count фаворизирует популярные треки. Transition-кандидатам присваивается `similarity_score=1.0` безусловно, они всегда "побеждают" KNN.
**Рекомендация:** Нормализация весов (PMI или TF-IDF аналогия). Скорректировать score transitions пропорционально их весу.
**Сложность:** Средняя

### 2.4 Popularity feedback loop в POPULAR strategy
**Источник:** Пользователь-одиночка
**Суть:** `list_popular` сортирует по `play_count DESC`. Top-20 треков окаменевают -- больше показов -> больше прослушиваний -> ещё больше показов.
**Рекомендация:** Time decay (`play_count * decay_factor(age)`) или play_count за последние N дней.
**Сложность:** Средняя

## 3. РЕКОМЕНДАЦИИ: UX (высокий приоритет)

### 3.1 Рекомендации не обновляются автоматически
**Источник:** Все пользовательские эксперты, UX-дизайнер
**Суть:** `fetchRecommendations` вызывается только при смене участника или переключении вкладки. После пения трека рекомендации "залипают". Нужно переключиться с вкладки и обратно.
**Фикс:** Автообновление после добавления трека в очередь + кнопка "Показать другие".
**Сложность:** Низкая

### 3.2 Лейблы стратегий -- криптичные
**Источник:** UX-дизайнер
**Суть:** "ПОПУЛЯРНОЕ", "В ВАШЕМ СТИЛЕ" -- технические метаописания. Нет связи с конкретными треками ("потому что вам понравилось X").
**Рекомендация:** Расширить `RecommendationResponse`:
```json
{
  "strategy": "last",
  "strategy_context": {
    "based_on_tracks": [{"title": "Группа крови", "artist": "Кино"}]
  }
}
```
Лейбл: `Похоже на "Группа крови" — Кино`
**Сложность:** Средняя

### 3.3 Нет negative feedback
**Источник:** UX-дизайнер, Пользователь-одиночка
**Суть:** Нет способа сказать "не хочу это направление". Единственный выход -- поиск.
**Рекомендация:** Dismiss-кнопка на карточке. На текущем этапе -- хотя бы запрашивать 20 треков, показывать 6, при dismiss показывать следующую.
**Сложность:** Низкая (фронтенд-only)

### 3.4 12 карточек -- перегрузка выбора
**Источник:** UX-дизайнер
**Суть:** 12 одинаковых карточек с gradient-placeholder требуют скролла. Парадокс выбора.
**Рекомендация:** 6 карточек + кнопка "Ещё" / "Перемешать".
**Сложность:** Низкая

### 3.5 TrackCard не кликабельна целиком
**Источник:** UX-дизайнер
**Суть:** onClick только на кнопке "ВЫБРАТЬ", хотя вся карточка имеет hover-эффект.
**Фикс:** Обернуть в ButtonBase.
**Сложность:** Минимальная

### 3.6 Нет фильтра по языку
**Источник:** Все пользовательские эксперты, Эксперт по музыке
**Суть:** 17000 русских+английских треков, нет способа указать предпочтение. Критично для компаний с разными языковыми предпочтениями.
**Рекомендация:** Chip-фильтры `[Все] [Русский] [English]` над сеткой рекомендаций. На бэкенде -- payload-фильтр в QDrant.
**Сложность:** Средняя

### 3.7 Нет прогресс-индикатора "обучения"
**Источник:** UX-дизайнер, Малая группа
**Суть:** Пользователь не знает, что его выбор влияет на рекомендации. Нет мотивации "спеть ещё для лучших рекомендаций".
**Рекомендация:** 3-сегментный индикатор: 1 трек -> 2 трека -> 3+ -> "Персональный микс".
**Сложность:** Низкая

## 4. МУЗЫКАЛЬНАЯ ЛОГИКА (средний приоритет)

### 4.1 Потеря временной динамики в аудио-фичах
**Источник:** Эксперт по музыке
**Суть:** Все фичи усредняются `mean(axis=1)`. Трек с тихим интро и мощным припевом = трек, который весь ровно громкий. "Stairway to Heaven" и "Highway to Hell" получат одинаковое среднее RMS.
**Рекомендация:** Добавить `std(axis=1)` для MFCC (13) и RMS (1) -- +14 измерений (итого 59-D).
**Влияние:** Различение монотонных треков от динамичных
**Сложность:** Средняя (потребуется переиндексация)

### 4.2 EMA alpha=0.3 -- инертность при смене жанра
**Источник:** Эксперт по музыке, Пользователь-одиночка
**Суть:** После 3 рок-треков человек решил спеть попсу -- нужно 3-4 попсовых подряд, чтобы система "переключилась". В караоке люди осознанно меняют настроение.
**Рекомендация:** Гибридная стратегия SESSION_AVG: `0.5 * KNN(последний_трек) + 0.5 * KNN(портрет)`. Или адаптивный alpha.
**Сложность:** Средняя

### 4.3 Нет учёта вокальной сложности и диапазона
**Источник:** Эксперт по музыке
**Суть:** Система не учитывает, удобно ли человеку петь рекомендуемый трек. Тенор получит рекомендацию Высоцкого (бас-баритон) просто по похожести звучания.
**Рекомендация:** Из вокальной дорожки (после UVR) извлечь min/max ноту и тональность. Использовать как payload-фильтр, не как часть вектора.
**Сложность:** Высокая (новый этап пайплайна)

### 4.4 Popularity boost в KNN-стратегиях
**Источник:** Эксперт по музыке
**Суть:** В KNN нет коррекции на узнаваемость. Для караоке люди предпочитают петь песни, которые знают.
**Рекомендация:** Мягкий boost: `fused += 0.05 * normalized_play_count`.
**Сложность:** Низкая

### 4.5 Межъязыковые рекомендации -- нет мягкого bias
**Источник:** Эксперт по музыке
**Суть:** Аудио-фичи языко-агностичны (рок = рок на любом языке). Система рекомендует английское после русского рока. Это хорошо для discovery, но в реальном караоке люди часто хотят оставаться в одном языке.
**Рекомендация:** Мягкий language bias: если последние N треков на русском, +0.1 к скору русскоязычных кандидатов.
**Сложность:** Низкая

## 5. ГРУППОВЫЕ СЦЕНАРИИ (средний приоритет)

### 5.1 Нет общих рекомендаций для группы
**Источник:** Все пользовательские эксперты, UX-дизайнер
**Суть:** Рекомендации строго персональные. Нет "что всем нам может понравиться". Стратегия GROUP_AVG (усреднение портретов) решила бы это.
**Рекомендация:** Кнопка "Для всех" в ParticipantSelector -> усреднение портретов участников.
**Сложность:** Средняя

### 5.2 Нет дуэтов и группового пения
**Источник:** Большая группа
**Суть:** `QueueEntry` содержит ровно один `participant_id`. Нельзя петь вдвоём.
**Рекомендация:** Расширить модель до `participant_ids: list[str]`. При finish -- обновить портреты всех.
**Сложность:** Высокая (изменение модели, API, UI)

### 5.3 Нет round-robin очереди
**Источник:** Большая группа
**Суть:** FIFO без гарантии честности. Один активный человек может захватить очередь. При 10 участниках -- до 31 мин ожидания.
**Рекомендация:** Опциональный round-robin: каждому по одному треку, прежде чем кто-то добавит второй.
**Сложность:** Средняя

### 5.4 Нет кнопки "Добавить участника" на рабочем экране
**Источник:** Большая группа
**Суть:** Для добавления опоздавшего нужно вернуться на SessionPage.
**Фикс:** Модальный диалог на QueuePage.
**Сложность:** Низкая

### 5.5 Кросс-участниковая дедупликация рекомендаций
**Источник:** Малая группа
**Суть:** Если Маша спела "Группу крови", Пете её всё ещё рекомендуют. Исключаются только свои спетые.
**Рекомендация:** Опционально исключать из рекомендаций треки, уже находящиеся в очереди сессии.
**Сложность:** Низкая

## 6. UX: КОНТЕКСТ ИСПОЛЬЗОВАНИЯ (средний приоритет)

### 6.1 Touch targets слишком мелкие
**Источник:** UX-дизайнер
**Суть:** Кнопка "ВЫБРАТЬ" ~60x28px, аватары ParticipantSelector 24x24px. Apple HIG: минимум 44px. Целевая аудитория -- подвыпившие люди в темноте.
**Фикс:** Увеличить все touch targets до 48px минимум.
**Сложность:** Низкая

### 6.2 Нет responsive layout
**Источник:** UX-дизайнер
**Суть:** Левая панель жёстко 480px. На телефонах layout ломается. На планшетах правая панель получает ~288px -- критически мало.
**Рекомендация:** Breakpoints: < 768px (одна колонка), 768-1024px (320px + flex), > 1024px (текущий).
**Сложность:** Высокая

### 6.3 ParticipantSelector: 4 градиента на 10 участников
**Источник:** Большая группа
**Суть:** 4 цвета аватаров -- при 10 людях каждый цвет повторяется 2-3 раза.
**Фикс:** Увеличить до 8-10 уникальных градиентов.
**Сложность:** Минимальная

### 6.4 Нет автопереключения ParticipantSelector
**Источник:** Малая группа
**Суть:** Когда очередь переходит к другому участнику, selector не переключается автоматически. Можно случайно добавить трек "от чужого имени".
**Фикс:** Автопереключение на `currentEntry.participant_id`.
**Сложность:** Низкая

### 6.5 Нет подтверждения "от чьего имени добавляется трек"
**Источник:** Малая группа
**Суть:** Snackbar: "Трек добавлен в очередь!" без имени. Можно ошибочно добавить от чужого имени, испортив чужой портрет.
**Фикс:** "Трек добавлен для [Вася]!" + визуальная привязка цвета рекомендаций к аватару.
**Сложность:** Низкая

### 6.6 Контрастность вторичных элементов
**Источник:** UX-дизайнер
**Суть:** `rgba(255,255,255,0.35)` -- контрастность ~2.5:1. Не проходит WCAG AA. В темноте караоке -- нечитаемо.
**Фикс:** Минимум `rgba(255,255,255,0.55)`.
**Сложность:** Минимальная

## 7. АРХИТЕКТУРА И OBSERVABILITY (низкий приоритет, но стратегически важно)

### 7.1 Нет логирования показанных vs выбранных рекомендаций
**Источник:** Архитектор, Эксперт по рекомендательным системам
**Суть:** Невозможно измерить качество рекомендаций. Нет CTR, precision@k, coverage.
**Рекомендация:** Добавить `recommendation_batch_id` в ответ и `source` в play_history.
**Сложность:** Средняя

### 7.2 Нет бэкапа QDrant
**Источник:** Архитектор
**Суть:** Transitions невосстановимы при потере данных. Вектора можно переиндексировать.
**Рекомендация:** Cron job для QDrant snapshots.
**Сложность:** Низкая

### 7.3 session_id не используется в get_recommendations
**Источник:** Эксперт по рекомендательным системам
**Суть:** Параметр принимается, но не задействован. Мог бы использоваться для кросс-участниковой фильтрации.

### 7.4 Поле completed в play_history не обновляется
**Источник:** Эксперт по рекомендательным системам
**Суть:** Всегда записывается как 0. Нет различия "допел до конца" vs "скипнул".

## 8. ИДЕИ НА БУДУЩЕЕ (бэклог)

| Идея | Источник | Ценность |
|------|----------|----------|
| Кнопка "Мне повезёт" (автовыбор лучшего) | UX | Снижает friction до 1 тапа |
| Превью трека (30 сек) | UX | Снижает неопределённость при выборе |
| Мини-профиль "Ваш вкус" (визуализация портрета) | UX | Gamification |
| Голосование за следующий трек | Большая группа | Социальный UX |
| "Предложить другу" (social endorsement) | Малая группа | Социальный UX |
| Контекстные фичи (время суток, день недели) | RecSys эксперт | Улучшение cold start |
| Multi-armed bandit для exploration/exploitation | RecSys эксперт | Алгоритмическая точность |
| WebSocket вместо поллинга очереди | UX, Архитектор | Снижение задержки |
| Настроение зала (session-level портрет) | Эксперт по музыке | Контекстная релевантность |
| Адаптивные веса fusion (жанрозависимые) | Эксперт по музыке | Точность для рэпа/поп |

## ЧТО ПОНРАВИЛОСЬ ЭКСПЕРТАМ (единогласно)

1. **Каскад стратегий** -- плавный переход от POPULAR к SESSION_AVG через промежуточные стадии. Участник не замечает "переключения".
2. **Граф переходов** -- захватывает реальные караоке-паттерны ("после грустной -- весёлую"), которые невозможно вывести из фич.
3. **Fallback-каскады** -- ни один пользователь не останется без рекомендаций, даже при отсутствии данных или падении QDrant.
4. **30% random в POPULAR** -- борьба с пузырём популярности из коробки.
5. **Вкладка "Поиск"** -- escape hatch для тех, кто знает, что хочет. Рекомендации не становятся тюрьмой.
6. **Dual-vector fusion** -- аудио определяет "вайб", лирика добавляет тематическую близость.
7. **Z-score нормализация** -- правильно решённая проблема доминирования масштаба.
8. **Чистая архитектура кода** -- хорошие docstrings, грамотные asyncio-паттерны, 540+ тестов.

# Architecture Decision Records (ADR)

## Формат записей

### ADR-NNN: Название решения
- **Статус:** Предложено / Принято / Отклонено / Заменено
- **Контекст:** Почему возник вопрос
- **Решение:** Что решили
- **Обоснование:** Почему именно так
- **Последствия:** Что это влечет

---

## Решения

### ADR-001: Единый стек Python + FastAPI
- **Статус:** Принято
- **Контекст:** В experiments бэкенд написан на Go (Gin), но мастер-промпт указывает Python. Два стека усложняют поддержку.
- **Решение:** Весь бэкенд пишем на Python с FastAPI. Go-код из experiments используем только как референс логики.
- **Обоснование:** Единый стек упрощает разработку и деплой. Python — основной язык для ML/аудио-обработки (librosa, whisperX и т.д.), нет необходимости в межъязыковых вызовах.
- **Последствия:** Нужно переписать логику Go-сервисов (recommendation_service, backend, audio_uploader) на Python.

### ADR-002: Гибридный ASR — Sonoix + WhisperX
- **Статус:** Принято
- **Контекст:** Нужна послоговая разметка для караоке. Sonoix — лучший по качеству для RU/EN, но платный. WhisperX — бесплатный, но хуже для русского.
- **Решение:** Sonoix API для пользовательских загрузок (online, <30 сек), WhisperX для первоначального бутстрапа каталога (5000-10000 треков).
- **Обоснование:** Экономия: Sonoix только для real-time обработки. Бутстрап — разовая операция, качество WhisperX приемлемо при наличии текстов из lrc-lib.
- **Последствия:** Два ASR-пайплайна, абстракция для подменяемости. Нужен API-ключ Sonoix (есть).

### ADR-003: QDrant + SQLite
- **Статус:** Принято
- **Контекст:** Много векторов (аудиофичи, эмбеддинги лирики, кластеры). Мало реляционных связей. Мастер-промпт предлагает QDrant.
- **Решение:** QDrant для всех векторных операций (рекомендации, поиск по похожести). SQLite для метаданных (треки, артисты, пути, сессии).
- **Обоснование:** QDrant — специализированная векторная БД с быстрым ANN-поиском, фильтрацией по payload. SQLite — zero-config, денормализованная схема, минимум ресурсов.
- **Последствия:** Два хранилища, синхронизация по track_id. QDrant поднимается в Docker. SQLite — файл на диске.

### ADR-004: React + TypeScript для фронтенда
- **Статус:** Принято
- **Контекст:** В experiments уже есть React-приложение с MUI. Нужно переделать под новые требования (сессии, очередь, послоговая подсветка).
- **Решение:** Продолжаем на React + TypeScript + MUI. Берем структуру из experiments как референс.
- **Обоснование:** Есть наработки, зрелая экосистема, хорошо подходит для SPA.
- **Последствия:** Фронтенд нужно переработать существенно: добавить сессии, очередь, послоговый плеер.

### ADR-005: Сессионная модель без регистрации
- **Статус:** Принято
- **Контекст:** Мастер-промпт требует: нет регистрации, нет user ID, рекомендации на основе анализа взаимодействия в рамках сессии.
- **Решение:** Сессия караоке-комнаты → подсессии участников (имя/никнейм). Рекомендации строятся на основе истории взаимодействия в подсессии. Никаких персональных данных.
- **Обоснование:** Минимальный UX-барьер. Соответствует модели использования караоке-клуба: пришли, попели, ушли.
- **Последствия:** Вектор "портрет пользователя" — временный, живёт в рамках сессии. Коллаборативные рекомендации — на основе кластеров агрегированных портретов (без привязки к людям).

### ADR-006: Деплой через Docker Compose
- **Статус:** Принято
- **Контекст:** Мастер-промпт требует "разворачивалось легко и просто, желательно в одну команду".
- **Решение:** Docker Compose как основной способ деплоя. Все сервисы в контейнерах.
- **Обоснование:** Один `docker-compose up` для запуска всего. Подходит для целевой аудитории (караоке-клубы, не highload).
- **Последствия:** Нужны Dockerfile для каждого сервиса, compose-файл с health checks и зависимостями.

### ADR-007: Без GPU — CPU-only или аутсорс
- **Статус:** Принято
- **Контекст:** Мастер-промпт: "Никаких GPU — либо аутсорс, либо вычисления на CPU".
- **Решение:** Sonoix (аутсорс) для ASR в runtime. WhisperX на CPU для бутстрапа. UVR (audio-separator) на CPU. librosa на CPU.
- **Обоснование:** Минимизация затрат на содержание. GPU не нужен при аутсорсе ASR.
- **Последствия:** Бутстрап будет медленнее (CPU whisperX), но это разовая операция.

### ADR-008: Деплой в России — VPN для внешних сервисов
- **Статус:** Принято
- **Контекст:** Сервис будет развёрнут в России. Некоторые внешние API (Sonoix, Docker Hub, PyPI, npm и т.д.) могут быть недоступны напрямую.
- **Решение:** Запросы к недоступным внешним сервисам проксировать через VPN. Доступны: HideMyName (ключ есть) и собственный сервер с Amnezia WG.
- **Обоснование:** VPN — готовое решение, не требует изменений в коде приложения. Достаточно настроить на уровне Docker-сети или хоста.
- **Последствия:** В Docker Compose может потребоваться VPN-контейнер (например, wireguard-клиент) для роутинга трафика worker-сервиса к Sonoix API. Документировать какие именно сервисы требуют VPN. При бутстрапе VPN не нужен (всё локально).

### ADR-009: lrc-lib — локальный дамп вместо API
- **Статус:** Принято
- **Контекст:** lrc-lib предоставляет публичный дамп базы текстов песен. Ходить в сеть за каждым треком при бутстрапе 5000-10000 песен — медленно и ненадёжно.
- **Решение:** Скачиваем дамп lrc-lib, используем локально при бутстрапе. После бутстрапа дамп удаляем.
- **Обоснование:** Экономия времени и сетевого трафика. Нет зависимости от доступности API. Нет лимитов на запросы.
- **Последствия:** Bootstrap CLI принимает параметр `--lrclib-dump /path/to/dump`. Поиск по дампу: SQLite-импорт или простой CSV/JSON-поиск по artist+title. После завершения бутстрапа дамп можно безопасно удалить.

### ADR-010: Реструктуризация — перенос реализации в v2/
- **Статус:** Принято
- **Контекст:** Директория `experiments/` с прошлыми наработками (Go backend, React frontend, Go recommendation service) находилась в корне репозитория рядом с текущей Python-реализацией. Это создавало путаницу в контексте при работе с кодом.
- **Решение:** Вся текущая реализация (backend/, worker/, shared/, tests/, docker-compose.yml, .env.example) перемещена в директорию `v2/`. Директория `experiments/` удалена. Документация (journals/, design/, master-promt.md) остаётся в корне.
- **Обоснование:** Чёткое разделение документации проекта и исходного кода. Эксперименты уже были полностью проанализированы в Фазе 1 (результаты в ADR-001 через ADR-009). Все относительные пути внутри v2/ (Dockerfile COPY, conftest.py, docker-compose build context) остаются рабочими без изменений.
- **Последствия:** Запуск тестов: `cd v2 && pytest tests/`. Docker: `cd v2 && docker compose up`. Все фазы 8a+ работают в `v2/`.

### ADR-011: Удаление видеогенерации, Feature Extraction на оригинале
- **Статус:** Принято
- **Контекст:** VideoGenerator генерировал MP4 с ASS-субтитрами для караоке. Фронтенд теперь рендерит слоги в реальном времени с подсветкой — серверный MP4 больше не нужен. Feature Extraction извлекал аудиофичи из instrumental (без голоса), но для рекомендаций нужны характеристики полного трека с вокалом.
- **Решение:** VideoGenerator полностью удалён (код, тесты, импорты). Feature Extraction теперь работает с оригинальным MP3 (с голосом). `clip_path` остаётся nullable полем в БД для обратной совместимости.
- **Обоснование:** Мёртвый код усложняет поддержку. Аудиофичи с вокалом точнее описывают трек для content-based рекомендаций (тембр голоса, мелодия).
- **Последствия:** Стриминг отдаёт instrumental (караоке) или mp3 (fallback). Образ bootstrap меньше (без FFmpeg для видео). При переиндексации QDrant нужно пересчитать 45-d векторы.

### ADR-012: Syllabify-then-align и LRCLib SQLite адаптер для бутстрапа
- **Статус:** Принято
- **Контекст:** Бутстрап-пайплайн получал слова из WhisperX, затем делил на слоги пропорционально по символам — грубая оценка таймингов. LRC-дамп lrclib на VPS — 78GB SQLite, а существующий `LRCLibDump` ожидает JSON-lines и грузит всё в RAM.
- **Решение:** Новый порядок: pyphen split текст → WhisperX force_align(слоги) → точные таймстемпы каждого слога из аудио. Новый адаптер `LRCLibSQLiteAdapter` — read-only запросы к SQLite на диске через `artist_name_lower` индекс. CLI: `--lrclib-sqlite /path/to/db`.
- **Обоснование:** WhisperX выравнивает каждый слог по реальному аудио — точнее, чем пропорциональное деление по символам. SQLite адаптер не грузит 78GB в RAM, использует OS page cache.
- **Последствия:** Бутстрап даёт более точные послоговые таймстемпы. Fallback (нет LRC) — старый путь через full ASR + пропорциональное деление.

### ADR-013: BS-Roformer для бутстрапа, MDX-NET для продакшна
- **Статус:** Принято
- **Контекст:** UVR-MDX-NET-Voc_FT.onnx (ONNX, SDR ~8-9) давал заметный вокальный bleed в инструменталах — особенно на треках с плотным женским вокалом. BS-Roformer-Viperx-1297 (PyTorch, SDR 12.9) — SOTA на SDX'23, значительно лучше качество. Но ~60 сек/трек на GPU vs ~16 сек у MDX-NET, и не работает быстро на CPU.
- **Решение:** Бутстрап использует BS-Roformer по умолчанию (`--uvr-model model_bs_roformer_ep_317_sdr_12.9755.ckpt`). Продакшн-воркер (Docker, CPU) пока остаётся на MDX-NET. `UVRSeparator` параметризован через `model_name`, обратная совместимость сохранена.
- **Обоснование:** Бутстрап — разовая операция на GPU-машинах. Разница 60 vs 16 сек/трек при 4800 треках на двух GPU — допустимые +3 часа ради значительно лучшего качества. Продакшн: решение по модели отложено (API-сервисы типа LALAL.AI ~$0.28/трек как опция).
- **Последствия:** Модель BS-Roformer (~639MB .ckpt) скачивается автоматически при первом запуске. Файлы инструменталов содержат имя модели в имени файла. При переходе продакшна на BS-Roformer потребуется GPU на сервере или API-интеграция.

### ADR-014: Multi-GPU bootstrap с preemptible-safe design
- **Статус:** Принято
- **Контекст:** Бутстрап ~4800 треков на одной GPU (RTX 4060) занял бы ~80 часов. Аренда GPU-сервера на Selectel (4×RTX 4090, preemptible — 4× дешевле) сокращает время, но сервер может быть прерван в любой момент (max 24ч).
- **Решение:** Новый режим `_run_local_gpu()` с `--gpu-id N`: atomic file claiming через `Path.rename()` в `.processing/` subdir, per-track QDrant flush (а не batch per 100), SQLite `timeout=30` для concurrent writers, `failed_ids: set` для пропуска ранее упавших треков. Launch script `run-gpu-server.sh` с auto-detect GPU count и stale claim recovery.
- **Обоснование:** Atomic rename на ext4 = безопасный concurrent claiming. Per-track flush гарантирует, что при прерывании теряется максимум 1 трек на воркер. Несколько воркеров на GPU (2-3) утилизируют GPU на 93-100% вместо 0-20% (CPU-bound librosa/ffmpeg шаги заполняют паузы между GPU-шагами).
- **Последствия:** 12 воркеров на 4×RTX 4090 дают ~8 треков/мин (×2.9 от 4 воркеров). Полный каталог ~4726 треков обрабатывается за ~10 часов. GPU RAM: 5-12GB из 24GB на GPU (запас для 3 воркеров).

### ADR-015: Нормализация аудиофичей при пользовательских загрузках
- **Статус:** Принято
- **Контекст:** Bootstrap-каталог прошёл через reindex_audio_features.py: raw L2 → z-score (per-dim mean/std) → L2-renorm. Но Worker создавал FeatureExtractor() без normalization_stats_path. Пользовательские загрузки оказывались в другом нормализационном пространстве (L2 only). Cosine similarity между каталожными (z-scored) и загруженными (raw L2) векторами — бессмысленна. Каскадный эффект: portrait участника портится при EMA-смешивании векторов из разных пространств.
- **Решение:** Worker получает `NORMALIZATION_STATS_PATH` через env var → передаёт в `FeatureExtractor(normalization_stats_path=...)`. JSON со stats генерируется reindex-скриптом и монтируется в контейнер (/data/models/). Одновременно исправлены 2 связанных бага: (1) update_portrait стирал lyrics_portrait при игре трека без лирики — теперь SQLite update пропускает столбец если NULL; (2) fallback при отсутствующем portrait без гарда на len(history) — теперь каскадная деградация.
- **Обоснование:** Все векторы должны быть в одном нормализационном пространстве для корректной cosine similarity. Стоимость фикса — $0 (JSON файл уже генерируется reindex-скриптом, нужно только пробросить путь).
- **Последствия:** При деплое нужно убедиться, что feature_normalization_stats.json доступен worker-контейнеру. Файл генерируется однократно при reindex и остаётся стабильным (пока каталог не пересчитывается с нуля).

### ADR-016: Чистый ре-бутстрап вместо патчинга
- **Статус:** Принято
- **Контекст:** После удаления GPU-сервера были потеряны QDrant-векторы (audio_features + lyrics_embeddings) для 4725 треков. Варианты: (1) патчинг — восстановление из stems + импорт между базами + фикс путей + фикс FTS; (2) чистая пересборка.
- **Решение:** Чистый ре-бутстрап каталога на новом GPU-сервере. Код подготовлен для прямой записи в production-базы: `--container-media-prefix` в CLI перезаписывает `instrumental_path` на контейнерный путь, DB инициализируется полной схемой из `init.sql` (FTS-триггеры срабатывают при INSERT), QDrant-коллекции создаются скриптом `init-qdrant.py`. Деплой через `docker-compose.prod.yml` с bind mounts вместо named volumes.
- **Обоснование:** Патчинг требовал множества ручных шагов с высоким риском ошибки (разные пути в старой/новой БД, отсутствие FTS, необходимость пересборки QDrant из stems). Чистый подход гарантирует консистентность и занимает ~10 часов на 4×RTX 4090.
- **Последствия:** Нужен временный GPU-сервер для ре-бутстрапа (~$5-10). Все данные (SQLite + QDrant) готовы к production-деплою без постобработки. `docker-compose.prod.yml` обеспечивает bind mounts для серверов с данными на root-диске.

### ADR-017: CTC Hybrid для слоговой разметки при загрузке треков
- **Статус:** Принято
- **Контекст:** Эксперимент M3 (5 треков, 7 методов) показал, что текущий ASR-based pipeline даёт плохие тайминги при отсутствии LRC (MAE 4.5с, hit rate 56%). WhisperX force_align (V2) требует GPU и двух проходов ASR. Sonoix+LLM коррекция не работает (MAE 13-54с). CTC Forced Aligner (MMS-300m ONNX) с гибридным word+char подходом показал лучшие результаты: MAE 0.240с, hit rate 71%, один проход на CPU за ~22с.
- **Решение:** Новый флоу загрузки: UVR → VAD (librosa) на вокале → Whisper tiny (идентификация песни) → LLM поиск текста в интернете → CTC word-level align → CTC char-level align внутри слов → слоговые тайминги. ASR используется только для идентификации, не для таймингов. Текст берётся из проверенного источника (lyrics-сайты).
- **Обоснование:** CTC Hybrid не зависит от качества ASR, работает на CPU, даёт тайминги на уровне WhisperX force_align (71% vs 72% hit rate) без GPU. Ключевой инсайт: pyphen proportional split — главный bottleneck word-level методов; char-level CTC внутри слов решает эту проблему.
- **Последствия:** Новая зависимость `ctc-forced-aligner` (~300MB модель). Нужен LLM API для поиска текста (~$0.001/трек). Fallback при ненайденном тексте — запрос к пользователю. Односложные слова пропускают char-level (pre-check на количество emission frames).

### ADR-018: v3-rc1 — GPU worker на выделенном сервере
- **Статус:** Принято
- **Контекст:** v2 worker работал на CPU (MDX-NET UVR ~7 мин/трек, Sonoix для ASR). Нужен переход на GPU pipeline (BS-Roformer + faster-whisper + CTC) для качества и независимости от Sonoix API. Два варианта: rc1 (выделенный GPU-сервер, ~$50/мес) vs rc2 (дешёвый VPS + внешние API, ~$15/мес + $0.30/трек). Выбран rc1.
- **Решение:** Выделенный сервер Xeon E-2236 + Tesla T4 16GB + 32GB DDR5. Docker Compose с 4 контейнерами (QDrant, backend, worker, frontend). Worker образ на CUDA 12.1 + cuDNN 8 + Python 3.11. GPU passthrough через nvidia-container-toolkit. Модели предзагружаются при старте (entrypoint.sh → download_models.py).
- **Обоснование:** Полный контроль над pipeline, нет зависимости от внешних API (кроме OpenAI для идентификации ~$0.0005/трек). T4 16GB достаточно для BS-Roformer + faster-whisper. Фиксированная стоимость вместо per-track.
- **Последствия:** Требуется "Above 4G Decoding" в BIOS для T4 (PCI BAR allocation). Dockerfile сложный (7 шагов, ~15 зависимостей). sentence-transformers<3 и transformers<5 из-за совместимости с torch 2.3.1. QDrant v1.16.2+ для qdrant-client 1.17. Сервер v2 (155.212.182.210) остаётся для текущего production, v3-rc1 на 212.41.1.108.

### ADR-019: Миграция графа переходов из QDrant в SQLite
- **Статус:** Принято
- **Контекст:** Граф переходов (коллекция `transitions` в QDrant) хранил 45-D вектор, который никогда не использовался для KNN-поиска — только payload-фильтрация (`scroll_filtered` по `from_track_id`). Также `record_transition` использовал read-modify-write (retrieve_payload → increment → upsert) без блокировки, что создавало race condition при concurrent finish_entry.
- **Решение:** Перенести граф переходов в SQLite-таблицу `transitions(from_track_id, to_track_id, weight)` с composite PK. Запись перехода через `INSERT ON CONFLICT DO UPDATE SET weight = weight + 1` (атомарная операция, без race condition). QDrant-коллекция `transitions` не удаляется из существующих деплоев, но больше не создаётся и не используется.
- **Обоснование:** Устранение race condition, упрощение архитектуры (убрана зависимость от QDrant для переходов), экономия памяти (нет лишнего HNSW-индекса), упрощение бэкапов (одна SQLite БД). Потеря исторических данных о переходах из QDrant приемлема — граф быстро наполняется при реальном использовании.
- **Последствия:** Константы `COLLECTION_TRANSITIONS` и `TRANSITION_DIM` удалены. QDrant init создаёт только 2 коллекции (audio_features, lyrics_embeddings). Тесты обновлены. Существующие деплои требуют перезапуск backend для создания новой таблицы (init.sql с IF NOT EXISTS).

### ADR-020: Рекомендации v2 — кластерная система с тегами настроения
- **Статус:** Принято
- **Контекст:** Реальное тестирование выявило, что пользователи не хотят регистрироваться (ParticipantSelector), рекомендуются малоизвестные треки (нет popularity boost), 12 одинаковых карточек без обложек. Проведён мозговой штурм (RECOMMENDATIONS_V2_BRAINSTORM.md), принято решение полностью перестроить систему.
- **Решение:** 8-фазная перестройка (R0-R7): очистка старого кода → popularity scoring (5 категорий треков) → кластеризация каталога (K-Means, 15-20 вайб-кластеров) → теги настроения (100-200 тегов на кластеры) → новый алгоритм (автокластеры сессии, макс. 3, MMR λ=0.7, popularity re-ranking, exploration-слот) → фото артистов (Spotify API) → фронтенд v2 (без участников, теги, 5 карточек, язык) → очистка.
- **Обоснование:** Кластерный подход решает проблему "среднего портрета" для компании с разными вкусами. Теги дают ручное управление поверх автоматики. Popularity scoring гарантирует, что рекомендуются узнаваемые песни. MMR обеспечивает разнообразие в пятёрке.
- **Последствия:** Удалены: EMA-портреты, 4 стратегии (last/last_two/session_avg), transitions, ParticipantSelector, SessionPage. Добавлены: 4 новые таблицы (catalog_clusters, mood_tags, artists, transitions удалена), 5 скриптов (parse_karaoke_charts, cluster_catalog, create_mood_tags, fetch_artist_images). API изменён: session_id вместо participant_id, tag_id/language параметры. Требуется: запуск скриптов кластеризации и парсинга на production данных.

### ADR-021: CTC Hybrid вместо Sonoix+WhisperX (заменяет ADR-002)
- **Статус:** Принято
- **Заменяет:** ADR-002 (Гибридный ASR — Sonoix + WhisperX)
- **Контекст:** Эксперимент M3 (7 методов, 5 треков) показал, что Sonoix BPE-токены дают плохие тайминги при LLM-коррекции (MAE 13-54с). CTC Forced Aligner (MMS-300m ONNX) с гибридным word+char подходом — лучшие результаты: MAE 0.240с, hit rate 71%, CPU-only, ~22с/трек. При этом текст берётся не из ASR, а ищется через LLM (OpenAI gpt-4o-mini) — из проверенных источников (Genius, lyrics-сайты).
- **Решение:** Новый pipeline: UVR → VAD → Whisper tiny (только идентификация) → LLM поиск текста → CTC word-level align → proportional syllable split. Sonoix API полностью убран. WhisperX используется только при бутстрапе (если нужен).
- **Обоснование:** CTC Hybrid не зависит от качества ASR, работает на CPU, даёт тайминги на уровне WhisperX force_align. Единственный внешний API — OpenAI (~$0.001/трек).
- **Последствия:** Зависимость `ctc-forced-aligner` (~300MB модель). CTC запускается в subprocess для изоляции от GPU VRAM. Нет необходимости в Sonoix API ключе.

### ADR-022: GPU worker вместо CPU-only (заменяет ADR-007)
- **Статус:** Принято
- **Заменяет:** ADR-007 (Без GPU — CPU-only или аутсорс)
- **Контекст:** CPU-only worker с MDX-NET + Sonoix обрабатывал трек ~7 мин (UVR на CPU ~5-8 мин). Переход на CTC Hybrid (ADR-021) убрал Sonoix, но UVR на CPU оставался бутылочным горлышком. BS-Roformer (SDR 12.9) на GPU даёт значительно лучшее качество за ~60-90с.
- **Решение:** Два режима worker: (1) GPU — выделенный сервер с Tesla T4, BS-Roformer + faster-whisper локально; (2) API — CPU VPS + MVSEP API + OpenAI Whisper API. Выбор через `WORKER_MODE` env var. Docker Compose: `docker-compose.gpu.yml` overlay с nvidia-container-toolkit.
- **Обоснование:** GPU-режим: полный контроль, нет зависимости от внешних API (кроме OpenAI ~$0.0005/трек), фиксированная стоимость ~$50/мес. API-режим: дешевле в простое, но ~$0.15-0.30/трек при нагрузке.
- **Последствия:** Требуется nvidia-container-toolkit и "Above 4G Decoding" в BIOS для T4. Dockerfile сложный (CUDA 12.1 + cuDNN 8). sentence-transformers<3 и transformers<5 из-за совместимости.

### ADR-023: Код в корне репозитория вместо v2/ (заменяет ADR-010)
- **Статус:** Принято
- **Заменяет:** ADR-010 (Реструктуризация — перенос реализации в v2/)
- **Контекст:** Директория v2/ была создана для разделения с experiments/ (ADR-010). После удаления experiments/ и завершения v3-rc1 worker, код мигрировал в корень: backend/, worker/, frontend/, shared/ — все на верхнем уровне. Директория v2/ больше не существует.
- **Решение:** Весь код в корне. Запуск: `make up-gpu` / `make up-api`. Тесты: `pytest tests/`. Docker build context = корень.
- **Обоснование:** Один уровень вложенности проще для навигации. v2/ потерял смысл после удаления экспериментов.
- **Последствия:** Все пути в docker-compose.yml, Dockerfile, conftest.py обновлены. Старые ссылки на v2/ в документации заменены.

# Мозговой штурм: Рекомендации v2

**Дата начала:** 2026-03-19
**Статус:** В процессе обсуждения

## Контекст

Реальное тестирование с пользователями выявило фундаментальные проблемы:
- Участники (ParticipantSelector) — не поняли, не захотели пользоваться
- Рекомендации — слишком много (12), незаметны, не читают
- Нет визуального различия между карточками (нет обложек)
- Пели из-под одного участника → рекомендовались малоизвестные похожие треки вместо хитов

Вывод: нужна новая версия модуля рекомендаций, а не рефакторинг.

---

## Вопрос 1: Участники — убрать или упростить?

### Решение: УБРАТЬ ПОЛНОСТЬЮ

Участники убираются из системы рекомендаций и из обязательного UI flow. Очередь = список треков без привязки к пользователям. Причины:
- Реальные пользователи не захотели пользоваться, не поняли зачем
- В караоке-комнате с друзьями люди сами разбираются кто поёт — кто-то подпевает, кто-то пропускает, кто-то выходит
- Формализовать "чья очередь" в UI — только мешать
- Поле "кто поёт" — тоже лишний friction

---

## Вопрос 2: Рекомендации — общие на сессию или персональные?

### Решение: ОБЩИЕ НА СЕССИЮ

Раз нет участников — рекомендации общие на сессию. Один "вкусовой портрет комнаты", но не один усреднённый вектор (это даёт "ни рыба ни мясо"), а **набор автокластеров** по вайбам.

Проблема "среднего портрета" решена через кластеризацию: если компания 50/50 рок/рэп, система видит 2 кластера и рекомендует из обоих, а не среднее между ними.

## Вопрос 3: Архитектура рекомендаций v2

### Два уровня кластеризации

**Уровень 1: Кластеры каталога (предрассчитанные)**
- 17k треков кластеризуются по fused-векторам (аудио + лирика) → 15-20 кластеров
- Кластеры = вайбы, не жанры. "Гитарная меланхолия" (Кино + Nirvana + Земфира), "тяжёлый драйв" (Ария + Metallica + AC/DC), "хип-хоп агрессия" (50 Cent + Oxxxymiron) и т.д.
- Грустный рэп и весёлый рэп — разные кластеры. Кино и Metallica — разные кластеры, хотя оба "рок"
- На каждый кластер навешивается 5-10 тегов с креативными названиями: "Шальная императрица", "Костёр на даче", "Да пошёл он...", "Друзья", "Тут дети"
- Один кластер → много тегов. Разные "входы" в один вайб
- Итого 100-200 тегов на 15-20 кластеров
- Пересчёт кластеров по крону раз в сутки (учитывает новые загруженные треки)
- У каждого трека в каталоге гарантированно есть лирика (это караоке)

**Уровень 2: Автокластеры сессии (динамические)**
- При запросе рекомендаций достаём вектора всех спетых треков сессии из QDrant
- Жадная кластеризация по fused cosine similarity (аудио 0.7 + лирика 0.3):
  - Первый трек = первый кластер
  - Каждый следующий: cosine к ближайшему центроиду > 0.7 → в этот кластер, иначе → новый
  - Макс. 3 кластера. Если должен появиться 4-й → в ближайший существующий
  - Одиночки: мягкий вес 0.5, полноценный кластер (2+ трека) = вес 1.0
- Вычисляется на лету (50 векторов × попарные cosine = микросекунды)

### Стратегии рекомендаций

**0 треков → POPULAR**
- 5 хитов из каталога
- С diversity (MMR), чтобы покрыть разные вайбы
- Не просто топ-5 по play_count, а по 1 из разных кластеров каталога

**1+ трек → CLUSTER**
- 5 рекомендаций = 4 из автокластеров + 1 exploration
- 4 кластерных слота распределяются пропорционально взвешенному размеру кластеров (мин. 1 слот на кластер)
- Для каждого кластера: KNN по центроиду с popularity boost + MMR
- Exploration: популярный трек, максимально далёкий от всех центроидов (anti-KNN)
- При 1-2 треках будет 1 кластер — это нормально, 4 слота из него + 1 exploration

### Теги настроения (UI)

- Над рекомендациями показываются 6-8 тегов-чипов
- Показываются теги от кластеров каталога, которые НЕ покрыты автокластерами сессии
- "Покрытие" = cosine similarity между центроидом автокластера сессии и центроидом кластера каталога > 0.7
- При нажатии тега → рекомендации заменяются на 5 треков из кластера каталога этого тега (KNN по центроиду кластера + popularity boost)
- После выбора трека → возврат в авторежим, автокластеры пересчитываются с учётом спетого
- Теги ротируются: покрытые кластеры исчезают, на их место приходят другие

### Загруженные пользователем треки

- Обрабатываются воркером → получают аудио + лирика вектора → в QDrant
- При пении → попадают в play_history → участвуют в автокластеризации сессии сразу
- В кластеры каталога попадают при следующем крон-пересчёте (раз в сутки)
- До крона не рекомендуются другим сессиям (нет модерации — это правильно)

## Вопрос 4: Баланс popularity vs similarity?

### Решение: КАТЕГОРИИ ТРЕКОВ + RE-RANKING

Не плавное числовое значение, а фиксированные категории с весами:

```
Вечный хит              (weight = 1.0)  — в караоке-списках, стабильно в чартах
Ситуативный хит         (weight = 0.7)  — сейчас в чартах
Лучшее артиста          (weight = 0.7)  — самый известный трек исполнителя (одинаковый вес с сит.хитом)
Бывший ситуативный хит  (weight = 0.4)  — был в чартах, вышел, метка навсегда
Просто песня            (weight = 0.1)  — всё остальное
```

Если трек попадает в несколько категорий — берём максимальный вес, не складываем.

**Жизненный цикл трека:**
1. Добавлен → "просто песня" (0.1)
2. Попал в чарты → "ситуативный хит" (0.7)
3. Вышел из чартов → "бывший ситуативный хит" (0.4), навсегда
4. Стабильно в чартах + в караоке-списках → "вечный хит" (1.0)

**Формула re-ranking:**
```
final_score = similarity * (1 + category_weight)
```

Хит с similarity 0.85 побеждает просто песню с similarity 0.93.

**Источники popularity:**
- Вечные хиты: парсинг 10 караоке-списков (vkaraoke.org, hitlist.ru, Billboard karaoke, Lucky Voice и др.)
- Ситуативные хиты: чарты с hitmotop.com (крон раз в неделю). Трек в топ-10 любого чарта ИЛИ присутствует минимум в 3 чартах → ситуативный хит
- Лучшее артиста: трек артиста с наибольшим chart_count. Если у артиста один трек — он автоматически "лучшее"
- Play_count: копится от реальных прослушиваний, добавляется к итоговому score как `normalized_play_count`

**Итоговый popularity:**
```
popularity = category_weight + normalized_play_count
```

**Веса — конфиг, не архитектура.** Можно менять без изменения кода. При необходимости — ввести "режим голодания" (когда хиты исключены как спетые → показывать бывшие ситуативные → просто песни).

## Вопрос 5: Diversity (MMR)?

### Решение: MMR НА ФИНАЛЬНОМ ЭТАПЕ, λ = 0.7

MMR применяется после сбора кандидатов из всех кластеров + exploration. Не внутри каждого кластера отдельно, а ко всем 5 финальным трекам.

**Алгоритм:**
1. Из каждого кластера берём топ-N кандидатов по final_score (после re-ranking с popularity)
2. Из exploration берём топ-N кандидатов (anti-KNN + popularity)
3. Объединяем в общий пул (~15-20 кандидатов)
4. Жадно выбираем 5 треков с MMR:
   - Первый: лучший по final_score
   - Каждый следующий: `mmr = 0.7 * final_score - 0.3 * max_similarity_to_already_selected`
   - Трек, похожий на уже выбранные, получает штраф

**Параметр λ = 0.7:**
- Приоритет у relevance (70%), но повторяющийся вайб штрафуется (30%)
- При λ = 1.0 — чистый score, без diversity
- При λ = 0.5 — слишком агрессивный diversity, может протащить нерелевантное
- 0.7 — конфигурируемый параметр, можно менять

**Эффект:** из двух рок-баллад с близким score MMR выберет одну и протащит что-то с другой энергией. Пользователь видит разнообразие в первых 2-3 позициях.

## Вопрос 6: Фото артистов — откуда?

### Решение: SPOTIFY + ЯНДЕКС.МУЗЫКА + ПЛЕЙСХОЛДЕР

Пользователи жаловались на однообразные карточки. Нужны не обложки альбомов, а фото артистов — это то, что визуально различает карточки в караоке.

**Источники (в порядке приоритета):**
1. **Spotify API** (Get Artist) — основной. Возвращает фото в нескольких размерах. Требует Premium у владельца приложения (с февраля 2026). Покрывает ~80% каталога.
2. **Яндекс.Музыка** (неофициальный API) — fallback для русскоязычных артистов, отсутствующих в Spotify.
3. **Плейсхолдер** — первая буква имени артиста на фоне градиента. Цвет градиента можно привязать к кластеру трека.

**Когда парсить:**
- Бутстрап: массовый прогон по ~5000 уникальным артистам (~1 запрос/сек = ~1.5 часа). Разовая операция.
- Загрузка пользователем: фоновая джоба параллельно с обработкой трека. Если не нашла — плейсхолдер.
- Крон (опционально): переодически пробовать найти фото для артистов с плейсхолдерами.

**Хранение:** поле `artist_image_url` в таблице tracks или отдельная таблица artists(name, image_url). Картинки скачивать и хранить локально (не зависеть от внешних CDN).

## Вопрос 7: Фильтр по языку?

### Решение: ТОГЛ "ТОЛЬКО НА РУССКОМ"

Один switch-фильтр. Не три варианта (Русский / English / Все), а бинарный: включён → только русские треки, выключен → все языки.

Применяется ко всему: авторекомендации, теги, поиск. На бэкенде — payload-фильтр `language = "ru"` в QDrant KNN. Поле `language` уже есть в треках.

## Вопрос 8: "Обновить" / "не то"?

### Решение: НЕ НУЖНО

Теги настроения заменяют кнопку "обновить". Не нравятся рекомендации → нажми другой тег. Отдельная кнопка "покажи другие" — лишний UI-элемент.

## Вопрос 9: Объяснение рекомендаций?

### Решение: НЕ НУЖНО

Никаких "потому что вы пели X". Просто 5 треков. Пользователям в караоке не нужна прозрачность алгоритма — нужны хорошие песни.

### Фаза R1: Popularity scoring

**Цель:** Каждый трек получает категорию popularity для re-ranking.

**Задачи:**
- Парсинг 5-6 русских + 4-5 английских караоке-списков → маппинг на каталог по artist+title
- Парсинг чартов с hitmotop.com → текущие хиты
- Новые поля в SQLite: `popularity_category` (enum: eternal_hit / current_hit / former_hit / artist_best / regular), `chart_count`, `chart_last_seen`
- Скрипт определения "лучшее артиста" — для каждого артиста трек с наибольшим chart_count
- Крон-скрипт обновления чартов (раз в неделю)

**Логика категоризации:**
```
В караоке-списках + стабильно в чартах → eternal_hit (1.0)
Сейчас в чартах (топ-10 любого ИЛИ в 3+ чартах) → current_hit (0.7)
Лучший трек артиста → artist_best (0.7)
Был в чартах, вышел → former_hit (0.4)
Остальное → regular (0.1)
```

**Входные артефакты:** Каталог 17k треков в SQLite
**Выходные артефакты:** Каждый трек имеет popularity_category. Скрипт крона.
**Коммит:** `feat: add popularity scoring with chart parsing and track categories`

**Входные артефакты:** Вектора в QDrant (audio_features + lyrics_embeddings)

### Фаза R4: Новый алгоритм рекомендаций

**Цель:** Полностью новый RecommendationService с автокластерами, popularity и MMR.

**Задачи:**
- Автокластеризация сессии: жадная по fused cosine (порог 0.7, макс. 3, одиночки вес 0.5)
- Распределение 5 слотов: 4 пропорционально кластерам + 1 exploration
- Для каждого слота: KNN по центроиду кластера → re-ranking по popularity_category → MMR (λ=0.7)
- Exploration: anti-KNN (далёкий от всех центроидов) + popularity boost
- POPULAR стратегия (0 треков): 5 хитов с MMR diversity из разных кластеров каталога
- Исключение спетых треков
- Фильтр по языку (если тогл включён)
- API: `GET /recommendations?session_id=X&language=ru` → strategy + tracks

**Формула re-ranking:**
```
final_score = similarity * (1 + category_weight)
```

**MMR:**
```
mmr = 0.7 * final_score - 0.3 * max_similarity_to_selected
```

**Входные артефакты:** catalog_clusters (R2), popularity_category (R1), _fused_knn_search (из R0)
**Выходные артефакты:** Новый RecommendationService, тесты
**Коммит:** `feat: new cluster-based recommendation engine with popularity and MMR`

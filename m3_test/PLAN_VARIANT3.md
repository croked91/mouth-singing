# Вариант 3: Sonoix + LLM-коррекция текста

## Суть подхода

У нас есть:
- Вокальная дорожка (WAV/MP3)
- Правильный текст песни (без таймингов)

Идея: отправить аудио в Sonoix API, получить BPE-токены с послоговыми таймингами,
а затем использовать дешёвую LLM для сопоставления/коррекции текста Sonoix
с правильным текстом, сохраняя тайминги.

## Контекст проекта (для субагента)

### Как работает Sonoix в production (воркер)
Файл `v2/worker/app/pipeline/sonoix_client.py`:
1. Загружаем аудио в Sonoix API (`POST /v1/files`)
2. Создаём задачу транскрипции (`POST /v1/transcriptions`, model: `stt-async-v4`)
3. Полим статус до `completed`
4. Получаем результат: `{"text": "...", "tokens": [{"text": " лю", "start_ms": 12340, "end_ms": 12560}, ...]}`
5. Удаляем файл и транскрипцию (cleanup)

### Формат токенов Sonoix
Sonoix возвращает **BPE sub-word токены**:
- Каждый токен — часть слова (примерно слог)
- Токен, начинающийся с пробела (` лю`) — **начало нового слова**
- Токен без пробела (`бовь`) — **продолжение слова**
- У каждого токена есть `start_ms`, `end_ms`, `confidence`, `language`

Пример: слово "любовь" → два токена: `" лю"` (start=12340, end=12560) + `"бовь"` (start=12560, end=12800)

### Как Syllabifier обрабатывает BPE-токены
Файл `v2/shared/karaoke_shared/utils/syllabifier.py`:
- Детектирует BPE-формат по наличию пробелов в начале токенов
- В BPE-режиме просто конвертирует токены 1:1 в `SyllableTiming`
- Тайминги берутся напрямую из Sonoix — **никакого пропорционального деления**
- Это лучшее качество таймингов из всех подходов

### Проблема
Sonoix может неточно распознать текст:
- "Она растёт прям из земли" → "она растёт прям из зимли" (ошибка в одном слоге)
- "Трава подружится с огнём" → "права подружится с агнём" (ошибки в начале слов)
- Для караоке это критично — пользователь видит неправильный текст

### Шаг 3: LLM-коррекция текста (ключевой шаг)
Создать скрипт `m3_test/variant3/llm_correct.py`:

#### 3.1 Формирование промпта
Задача для LLM: получив правильный текст и список BPE-токенов от Sonoix,
вернуть **скорректированный список токенов**, где текст заменён на правильный,
а тайминги сохранены.

**ВАЖНО**: Не отправлять в LLM все токены трека сразу — это может быть 500+ токенов
(~3-5 минут песни). Обрабатывать **порциями по строкам** известного текста:
1. Сгруппировать Sonoix-токены в "слова" (по пробелам)
2. Грубо сопоставить Sonoix-слова с известными строками (по позиции — первые N слов = первая строка и т.д.)
3. Отправлять в LLM по 2-4 строки за раз с соответствующими токенами
Это уменьшит контекст, повысит точность и удешевит вызовы.

```python
prompt = f"""Ты — корректор текста для караоке-системы.

Тебе дан правильный текст песни и список токенов от системы распознавания речи (ASR).
Каждый токен — это часть слова (примерно слог) с таймингом.
Токен, начинающийся с пробела — начало нового слова.

Твоя задача: скорректировать текст токенов так, чтобы он совпадал с правильным текстом,
сохраняя при этом тайминги и структуру токенов (разбивку на части).

Правила:
1. Количество токенов может измениться (если ASR пропустил или добавил слова)
2. При добавлении нового токена — интерполируй его тайминг между соседними
3. При удалении лишнего токена — перераспредели его время между соседними
4. Старайся сохранить исходные тайминги максимально точно
5. Пробел в начале токена означает начало слова — сохраняй эту семантику
6. Возвращай ТОЛЬКО JSON массив токенов, без пояснений

ПРАВИЛЬНЫЙ ТЕКСТ:
{correct_lyrics}

ТОКЕНЫ ОТ ASR (JSON):
{json.dumps(sonoix_tokens, ensure_ascii=False)}

Верни скорректированный JSON массив токенов в формате:
[{{"text": " лю", "start_ms": 12340, "end_ms": 12560}}, ...]
"""
```

#### 3.2 Варианты промптов для тестирования
Подготовить **3 варианта промпта** с разной степенью детализации:
- **A (минимальный)**: только текст + токены, короткая инструкция
- **B (подробный)**: с примерами коррекции (few-shot)
- **C (двухэтапный)**: сначала LLM выравнивает слова, потом корректирует слоги

#### 3.3 Вызов LLM
```python
import anthropic  # или openai

client = anthropic.Anthropic(api_key=LLM_API_KEY)
response = client.messages.create(
    model="claude-haiku-4-5-20251001",
    max_tokens=4096,
    messages=[{"role": "user", "content": prompt}],
)
corrected_tokens = json.loads(response.content[0].text)
```

#### 3.4 Валидация ответа LLM
После получения ответа проверить:
- Все токены имеют text, start_ms, end_ms
- start_ms < end_ms для каждого токена
- Токены идут в хронологическом порядке (start_ms[i] <= start_ms[i+1])
- Конкатенация текстов токенов даёт правильный текст (без учёта пробелов/регистра)
- Если валидация не прошла — залогировать ошибку и пометить трек как failed

### Шаг 4: Конвертация в SyllableTiming
После LLM-коррекции токены имеют формат BPE (пробел = начало слова).
Конвертация — как в Syllabifier._from_bpe_tokens:
```python
syllable_timings = [
    SyllableTiming(syllable=t["text"], start=t["start_ms"]/1000, end=t["end_ms"]/1000)
    for t in corrected_tokens
    if t["text"].strip()
]
```

## Критерии успеха
- WER после LLM-коррекции < 2% (почти идеальный текст)
- MAE таймингов < 0.08с (Sonoix обычно точнее WhisperX)
- Стоимость обработки одного трека < $0.01 (для Haiku)
- LLM возвращает валидный JSON в > 90% случаев

## Потенциальные проблемы и решения
1. **LLM ломает тайминги** — вместо сохранения оригинальных таймингов, придумывает свои.
   Решение: в промпте явно запретить менять тайминги для совпадающих токенов; после ответа
   LLM восстановить оригинальные тайминги для токенов, текст которых не изменился.

2. **LLM не возвращает валидный JSON** — ответ содержит markdown/пояснения.
   Решение: использовать `response_format={"type": "json_object"}` (OpenAI) или
   парсить JSON из markdown-блока; retry с уточнением промпта.

3. **Sonoix пропускает целые фразы** — большой кусок текста без токенов.
   Решение: детектировать пропуски (большие разрывы во времени), для пропущенных
   фраз создать "заглушки" с интерполированными таймингами.

4. **Стоимость LLM слишком высока** — при 17000 треках.
   Решение: если качество Sonoix достаточно высокое (WER < 5%), можно использовать
   детерминистический алгоритм сопоставления (как в варианте 2, шаг 2.2) вместо LLM.
   Это отдельный суб-эксперимент: **вариант 3b — Sonoix + детерминистическая коррекция**.

5. **API rate limits** — при массовой обработке.
   Решение: для теста на 5 треках неактуально. Для продакшена — батчинг и задержки.

## Дополнительный суб-эксперимент: 3b (без LLM)
Если LLM-коррекция окажется дорогой или нестабильной, попробовать:
- Взять BPE-токены от Sonoix
- Сконкатенировать в слова (по пробелам)
- Сопоставить слова с известным текстом через `difflib.SequenceMatcher`
- Для совпавших слов — использовать оригинальные тайминги токенов Sonoix
- Для несовпавших — заменить текст, сохранив тайминги (пропорциональное деление по слогам pyphen)
- Это дешевле LLM, но может быть менее точно при сильных расхождениях

Алгоритм подробнее:
1. Группируем BPE-токены в слова: `[(" лю", 100, 200), ("бовь", 200, 350)]` → `{"word": "любовь", "tokens": [...], "start_ms": 100, "end_ms": 350}`
2. Строим два списка слов: `sonoix_words` и `known_words` (из текста)
3. `difflib.SequenceMatcher(None, sonoix_words_lower, known_words_lower).get_opcodes()`
4. Для `equal` — берём токены Sonoix как есть
5. Для `replace` — берём временной диапазон заменяемых Sonoix-слов, разбиваем known-слова на слоги через pyphen, распределяем время пропорционально
6. Для `insert` (слова в known, но не в Sonoix) — интерполируем тайминг
7. Для `delete` (лишние слова в Sonoix) — пропускаем, перераспределяем время
